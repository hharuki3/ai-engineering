00
PROLOGUE

同じモデルなのに、なぜ差が出るのかWhere the difference comes from

同じツール、同じモデル、同じ月額。それなのに、隣のチームは実装をまるごと任せて回しているのに、こちらは毎回チャットに張り付いて手直しをしている——この差は、モデルの賢さでは説明できない。

差を生んでいるのは、モデルの周りにあるものだ。何を読ませているか。何を強制しているか。失敗をどこに記録しているか。誰が次の指示を出しているか。

この資料では、その「周り」を設計する4つの技術を、内側から外側へ順番に歩いていく。キーワードは一つ。設計対象は、外へ。

01
THE MAP

4つの同心円Four concentric circles

4つの xx は流行の交代劇ではない。内側の円を土台に、外側へ設計対象が広がってきた積み上げだ。円をなぞってみてほしい——1つの文、渡す情報、実行環境、そしてプロセス全体へ。

① プロンプトHOW TO ASK

どう聞くか。1回の応答の質を上げる。設計対象は「1つの指示文」。

② コンテキストWHAT TO FEED

何を渡すか。モデルが今アクセスできる情報の全体を設計する。

③ ハーネスWHAT SURROUNDS

周りをどう作るか。ツール・強制・検証。モデル以外の全部。

④ ループWHO PROMPTS

誰が回すか。「プロンプトする」という行為自体を仕組みにする。

02
LAYER 1 — PROMPT ENGINEERING

職人芸の時代How to ask

この2年の主役。役割を与える、手順を分解させる、良い例と悪い例を見せる、出力の型を指定する。タスクごとに“効く一文”は違うから、探し当てる腕が問われた。それは価値ある職人芸だったし、今も土台であり続けている。

ただし限界もはっきりしている。プロンプトが最適化できるのは1ターンの応答まで。モデルはコンテキストの外を何も覚えていない。10ステップ動くエージェントに必要なのは「良い一文」ではなく、10ステップぶんの足場だ。

この層の設計単位：役割 ＋ 手順分解（step by step）＋ few-shot ＋ 出力フォーマット ＝ よく効く1つの指示文。ここで終わると、成果は毎回モデルの機嫌に左右される。

03
LAYER 2 — CONTEXT ENGINEERING

記憶は有限資源であるWhat to feed

問いを「どう言うか」から、モデルが今アクセスすべき情報の全体をどう組むかへ。システムプロンプト、会話履歴、検索結果、ツール定義、メモリ——窓に載るものすべてが設計対象になる。Claude Code なら CLAUDE.md がその代表格だ。

そして重要な事実がひとつ。コンテキストウィンドウは埋まるほど濁る。窓が混雑すると、重要な指示が埋もれ、推論の精度が落ちていく。いわゆる context rot。だからこの層の設計は「足す」より「引く」——スライダーで体感してみてほしい。

快適圏ウィンドウ使用率 18%
推論の安定度
96%

  • 何を残し、何を外すかが勝負。履歴の圧縮、不要ツールの排除、大きな出力のファイル退避
  • 必要なときだけ知識を載せる ＝ Skills の progressive disclosure（詳細は別紙2・3章）
“context engineering is in, and prompt engineering is out” — Gartner（2025年7月）。エージェント成功の最大の予測因子はモデル選定よりコンテキスト設計、という論調。出典 ↗
04
LAYER 3 — HARNESS ENGINEERING

馬ではなく、馬具を作るWhat surrounds the model

“harness” の原義は馬具。どれほど速い馬でも、頭絡と手綱がなければ目的地には着かない。馬（モデル）を育てるのは研究所の仕事。私たちの仕事は、馬具一式を設計することだ。

定義はシンプルに割り切れる。Agent ＝ Model ＋ Harness. モデルでない部分は、全部ハーネス。下の図の部品をクリックして、それぞれの役割を確かめてほしい。

HARNESS PARTS

部品を選んでください

左の図の6つの部品は、どれも「モデルに賢さを足す」ものではない。モデルの賢さを取りこぼさないための足場だ。

ヒント：タップ / クリックで切り替わります。

この層には、姿勢の転換がひとつ含まれている。エージェントが失敗したとき、「モデルがまだ賢くない」と言うのは簡単だ。しかしその失敗の多くは、規約に書いていない・強制していない・検証していない——つまり“skill issue”、設定の問題である。

だから運用はラチェット（逆戻り防止の爪）になる。失敗を1回見るたびに、CLAUDE.md か hook に1行刻む。同じ失敗は二度と起こらない。足場は失敗の数だけ強くなる。

効果の実例：Terminal Bench 2.0 で、モデルは同じままハーネスだけを作り込んで Top30 → Top5 へ。「良いモデル × 雑な足場」より「そこそこのモデル × 良い足場」が勝つ。
05
LAYER 4 — LOOP ENGINEERING

自分の仕事を、コードにするWho prompts the model

最後の円では、視点がもう一段外に出る。ハーネスを整えても、毎ターン人間が指示を打っているなら、ボトルネックは人間の指だ。ならば——「指示を出す」という行為そのものを仕組みにできないか？

“I don't prompt Claude anymore. I have loops running that prompt Claude. My job is to write loops.” — Boris Cherny（Anthropic / Claude Code 責任者）。発言クリップ（X）↗ ／ 解説記事（The New Stack）↗

ループを支えるのが外部メモリ（md ファイルや Linear）だ。「済み・未」を会話の外に置くことで、毎回ゼロから始めなくてよくなる。会話は消えるが、メモリは積み上がる。

ここまで来ると、人間の役割は明確になる。ループを書くこと。そして、ループが吐くものを検証する基準を書くこと。手を動かす場所が、また一段外へ移った。

06
INTERLUDE

レバレッジは移動するThe leverage keeps moving

ふりかえると、変わってきたのは技術そのものより「どこに力をかけると一番効くか」という点の位置だ。1文 → 情報 → 環境 → プロセス。いま費用対効果が最も高いのは、多くのチームにとってハーネスの設計。ループはその先にある。

そしてこれは終わらない。モデルが賢くなるたびに足場が不要になるのではなく、足場ごと、より高いタスクへ引っ越すだけだ。「賢い一文を探す人」から「賢く動く環境を設計する人」へ——役割の重心は、静かに、しかし確実に移っている。

07
CAUTION

設計なき自走は、ゴミを量産するThe garbage machine problem

ここまでの話には、強い注意書きが要る。ハーネスを作り込まずにループだけ回すと、それはトークンを燃やしながらミスを無人で量産する装置になる。しかも請求は正確に届く。

コスト暴走

ループもサブエージェントも各自が課金対象。並列 × 繰り返しで消費は一気に膨らみ、“回しっぱなし”は青天井になりうる。

Done の自己申告

verifier を分けないと「できました」が自己採点になる。動かないコードに、動いた分だけ課金される。

理解の空洞化

自分で書いていないコードが増えるほど、実態と理解の差が開く（comprehension debt）。ループが速いほど負債も速い。

丸投げへの退化

回り出すと出力をそのまま採用したくなる。設計者が「ボタンを押す人」になった時点で、品質の下り坂が始まる。

4つの安全装置（走らせる前に必ず）：① 予算と上限——トークン・実行回数・最大時間を先に決める ② Done 条件を検証可能な形で書く ③ 検証は人間が持つ——作る人と確かめる人を分ける ④ 小さく試してから広げる。設計する人であり続けることが、すべての前提。

HANDS-ON

明日の一歩 — 効き目の順にStart here tomorrow

理屈はもういい、どこから触るか。答えはハーネスから。難しい自動ループより先に、足場を3点だけ整える。CLAUDE.md はたとえばこの程度で十分に効く。

# プロジェクト規約（毎ターン読み込まれる）
## コマンド
- テスト:   pnpm test ／ Lint・型:  pnpm lint && pnpm typecheck
## 規約
- パッケージマネージャは pnpm 固定（npm / yarn 禁止）
- ログは logger を使う（console.log 禁止）
## 触ってはいけない
- /legacy 配下は変更しない ／ DB マイグレーションは手動確認後のみ
## Done の条件
- 対象テストが green ＋ lint / 型エラー 0

スターター・チェックリスト

0 / 7
  • CLAUDE.md を書く（10〜20行）コマンド・レッドライン・Done 条件だけ。60行を超えたら削る
  • hooks を2本入れるPreToolUse で危険コマンドをブロック ＋ PostToolUse で lint / 型を自動実行
  • コスト上限の場所を確認するサブスクなら /cost、API なら Console の spend limit を一度見ておく
  • 毎回説明している手順を SKILL.md に切り出すdescription は「いつ使うか」を退屈なほど具体的に
  • フェーズ別に CLAUDE.md を分ける要件定義は docs/spec/ 配下など、「場所」で切るのが原則
  • 検証可能な停止条件を書いてから小さく自走させる/goal に「test/auth 全 green かつ lint 0」のような形で
  • 失敗を見たら、その場で1行足すCLAUDE.md か hook へ。ラチェットは今日から回り始める

具体的な設定値・settings.json の全文・フェーズ別制御の詳細は、別紙 「実践ガイド — ハーネスの具体設定」 に。

EPILOGUE

持ち帰るのは4行でいいTakeaways

−結果がモデルの機嫌任せで、当たり外れが大きい
−同じ失敗を毎回、口頭で直させている
−毎ターン人が張り付き、レビューは自己申告
＋足場が品質を安定させ、モデル更新の恩恵もそのまま乗る
＋失敗はルール化され再発しない——足場は失敗の数だけ強くなる
＋検証付きで一部を任せ、人はループと基準の設計に回る

① 4つの xx は積み上げ——どう聞く → 何を渡す → 周りを作る → 誰が回す。
② レバレッジは「良い一文」から「良い環境」へ移動した。いま効くのはハーネス。
③ 設計なき自走はゴミ量産マシン。予算・Done 条件・人による検証をセットで。
④ 明日やるのは2つだけ：CLAUDE.md を短く書く、hooks で1つ強制する。
