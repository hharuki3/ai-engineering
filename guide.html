CLAUDE.md に何を書くか

プロジェクト直下に置くと毎ターン system prompt に載るメモリファイル。原則は「短く・理由付きで・過去の失敗だけをルール化」。目安は60行以内。

置き場所と読み込み

  • ./CLAUDE.md … リポジトリ直下。チーム共有（コミット対象）
  • ./CLAUDE.local.md … 個人用（gitignore 推奨）
  • ~/.claude/CLAUDE.md … 全プロジェクト共通の個人設定
  • サブディレクトリに置けば、そのフォルダ作業時のみ追加読み込み（→ 4章で活用）

書く中身（WHAT / WHY / HOW ＋ レッドライン ＋ Done）

「何を・なぜ・どうやって」＋「触ってはいけない」＋「完了条件」が揃うと、エージェントが迷わず・暴走しにくい。以下は Laravel プロジェクトでの抜粋。

# sunshine-server

## WHAT — 技術スタック / 構造
- Laravel 10 / PHP 8.1 / MySQL 8.4 / Redis / Vue 3、AWS ECS Fargate
- ローカルは Docker Compose（sunshine_app / _nginx / _mysql / _redis）
- app/: Http/Controllers/{Api,Backend} / UseCase / Services / Repositories
        / Models / ViewModels / Jobs / Enums
- routes/: api.php(モバイル) / admin.php(管理) / auth.php(認証)

## WHY — 目的
モバイルアプリ「マイサンシャインシティ」のバックエンド。
スタンプ / クーポン / お知らせ / イベント等を提供し、マチパス・Epark と連携。

## HOW — コマンド
- make up / work / down / format / openapi
- テスト: php artisan test [--testsuite=Unit] [--filter=ClassName]
- テストDB: sunshine_testing（初回のみ make create-testdb）

## 規約（アーキテクチャ）
- 流れ: Model → Repository → UseCase/Service → Controller → ViewModel
- 各層は Contract(interface) で DI、バインドは ServiceProvider
- 定数は Model に定義（config ではない）
- API 変更時はコントローラに #[OpenApi\Operation] 属性を付与
- 命名はキャメルケース（PSR-12、make format で強制）

## 規約（エラーハンドリング）
- ドメイン例外は app/Exceptions/ 配下に定義し UseCase/Service で throw
- try-catch は Controller 層のみ。ドメイン例外 → 最後に Exception の順で catch
- ログは Log::error()。重要操作は Log::withContext([...]) を付与
  （os キーは TraceLog ミドルウェアが自動付与、個別追加不要）

## 触ってはいけない（レッドライン）
- 生成物 docs/api/v1/openapi.json は手編集しない（make openapi で再生成）
- DB マイグレーションは手動確認後のみ実行
- 本番 / 開発DBに直接接続しない。.env / secrets は読まない・出力しない

## Done の条件
- 対象テストが green ＋ make format 済み
- API を変更したら make openapi を実行し差分を確認
コツ：ルールは「増やすのは失敗を見た時だけ／モデルが賢くなったら削る」。全部盛りにすると1行ごとの重みが下がり、かえって守られなくなる。詳細なアーキ解説など“辞書的”な内容は Skills / references に逃がし、CLAUDE.md は毎ターン効く要点に絞ると軽い。

hooks の使い方

CLAUDE.md が「お願い」なら、hooks は「強制」。エージェントのライフサイクルの特定点でシェル等を実行し、通す / 止めるを決める。.claude/settings.json に書けばチーム全員に効く。

主なイベント

イベントタイミング典型用途
PreToolUseツール実行の直前危険コマンドをブロック、承認ゲート
PostToolUseツール実行の直後編集後に lint / 型 / test、自動整形
UserPromptSubmit入力送信時入力の検査・付帯情報の注入
Notification通知発生時（承認待ち等）デスクトップ通知・音
Stop応答終了時完了チェック・通知

設定例（.claude/settings.json）― 実プロジェクト版（Laravel + Docker）

動作確認済みの構成。5つの hook を入れている。settings.json は標準 JSON でコメント不可なので、意味は下の箇条書きで補足する。

  • PreToolUse / Bash（危険コマンド）：rm -rf / dd / mkfs / DROP TABLE / chmod -R 777 などを検出し、実行前にブロック
  • PreToolUse / Bash（コミット前テスト）：git commit を含むコマンドの前に php artisan test を実行し、失敗したらコミットをブロック
  • PreToolUse / 編集（保護ブランチ）：master / develop への直接編集を禁止（feature ブランチを強制）
  • PostToolUse / 編集（自動整形）：.php を編集したら php-cs-fixer を自動実行
  • Notification / Stop：承認待ち・作業完了を macOS のデスクトップ通知（音付き）で知らせる
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 -c \"import sys,json;d=json.load(sys.stdin);cmd=d.get('tool_input',{}).get('command','');any(p in cmd for p in ['rm -rf /','rm -rf ~','dd if=','mkfs','DROP TABLE','chmod -R 777','truncate /']) and print(json.dumps({'decision':'block','reason':'危険なコマンドの実行をブロックしました。直接ターミナルで実行してください。'}))\""
          }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 -c \"import sys,json,subprocess as sp;d=json.load(sys.stdin);cmd=d.get('tool_input',{}).get('command','');'git commit' not in cmd or (lambda r:r.returncode==0 or print(json.dumps({'decision':'block','reason':'テストが失敗したためコミットをブロックしました:\\n'+(r.stdout+r.stderr)[-500:]})))(sp.run(['docker','exec','sunshine_app','php','artisan','test'],capture_output=True,text=True,timeout=300))\""
          }
        ]
      },
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "python3 -c \"import sys,json,subprocess as sp;d=json.load(sys.stdin);b=sp.run(['git','-C','/path/to/your-project','branch','--show-current'],capture_output=True,text=True).stdout.strip();b in ('master','develop') and print(json.dumps({'decision':'block','reason':b+'ブランチへの直接編集は禁止です。featureブランチで作業してください。'}))\""
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "python3 -c \"import sys,json,subprocess as sp;d=json.load(sys.stdin);f=d.get('tool_input',{}).get('file_path','');f.endswith('.php') and sp.run(['docker','ps','--filter','name=sunshine_app','--filter','status=running','-q'],capture_output=True).stdout.strip() and sp.run(['docker','exec','sunshine_app','./vendor/bin/php-cs-fixer','fix','--using-cache=no',f.replace('/path/to/your-project/','')],capture_output=True)\""
          }
        ]
      }
    ],
    "Notification": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "python3 -c \"import sys,json,subprocess as sp;d=json.load(sys.stdin);m=d.get('message','Claude Codeから通知')[:100];sp.run(['osascript','-e','display notification '+json.dumps(m)+' with title \\\"Claude Code\\\" sound name \\\"Glass\\\"'])\""
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "python3 -c \"import sys,json,subprocess as sp;d=json.load(sys.stdin);sp.run(['osascript','-e','display notification \\\"Claude Codeの作業が完了しました\\\" with title \\\"Claude Code\\\" sound name \\\"Glass\\\"'])\""
          }
        ]
      }
    ]
  }
}
設計原則：「成功は無言、失敗は饒舌」。通過時は何も返さず、失敗時だけ decision: block と reason を返すと、フィードバックがほぼ無料で、直すべき時だけ効く。hook は stdin に JSON（tool_input 等）を受け取る。

Skills（SKILL.md）の設定例

再利用する手順・知識を SKILL.md にまとめ、必要な時だけ context に読み込ませる仕組み。「同じ説明を毎回書く」をやめられる。

置き場所

  • .claude/skills/<name>/SKILL.md … プロジェクト共有（git でチーム配布）
  • ~/.claude/skills/<name>/SKILL.md … 個人・全プロジェクト共通

フォルダ構成

.claude/skills/pr-review/
  SKILL.md        # 必須：frontmatter(name/description)＋手順
  scripts/        # 任意：決定的な処理スクリプト
  references/     # 任意：必要な時だけ読むドキュメント
  assets/         # 任意：出力に使うテンプレ・アイコン等

SKILL.md の中身（例：PRレビュー用）

---
name: pr-review
description: PR差分をレビューする時に使う。観点（N+1・入力検証・
             例外処理）とコマンドをまとめたもの。
---

# PR レビュー手順
1. `git diff main...HEAD` で差分を確認する
2. 次の観点でチェックする
   - N+1 クエリ / インデックス漏れ
   - 入力バリデーション（zod があるか）
   - 例外処理（握りつぶしていないか）
3. 指摘は severity（high / mid / low）付きで箇条書き

## 詳細な観点
- references/security-checklist.md を参照
起動の肝は description：「いつ使うか」を具体的に書くほど自動起動の精度が上がる（賢い説明より“退屈で具体的”が勝つ）。name はフォルダ名と一致必須。本体は短くし、詳細は references/ に逃がすと、必要な時だけ context に載る（＝ progressive disclosure）。複数スキルや MCP をまとめて配布したい時は plugin にパッケージする。
Skills はどの層？　→ 実は「コンテキスト」でも「ハーネス」でもある。
・コンテキスト面：「何を・いつ渡すか」を制御する仕掛け。必要な時だけ知識を context に載せる progressive disclosure は、コンテキスト設計そのもの。
・ハーネス面：SKILL.md・scripts・plugin はモデルを包む足場の一部。再利用可能な“能力”として agent に組み込む＝ハーネスの構成要素。
・さらにループ面：ループの5部品の1つでもある。
→ Skills は層をまたぐ「橋渡し」の存在、と捉えると位置づけがすっきりする。

フェーズ別のコンテキスト制御

実装だけでなく要件定義〜設計でも使うなら、「毎ターン載る情報」をフェーズで出し分けたい。原則は内容を切り替えるのではなく「場所（ディレクトリ）」か「セッション」で切ること。

何が“毎ターン確実に”載るか

  • 確実に載るのは、セッションの作業ディレクトリ(cwd)から上へ辿って見つかる CLAUDE.md（/compact 後も再読込される）
  • サブディレクトリの CLAUDE.md は起動時には載らない。そのフォルダのファイルを読んだ時にロードされる（＝場所に紐づく）
  • @import は常時インライン展開（条件分岐にならず、トークンも減らない）。今ロード中の確認は /memory

方法1 — ディレクトリ単位で CLAUDE.md を分割

CLAUDE.md            # 共通の普遍情報だけ薄く（WHY / 地図 / レッドライン）
app/CLAUDE.md        # 実装フェーズの規約（アーキ・エラー処理・命名）
docs/spec/CLAUDE.md  # 要件定義フェーズ専用
  • 実装の重い規約をルートから外し、app/ 配下でだけ載るようにする
  • 要件は docs/spec/ の成果物を触りながら進めれば、要件用 CLAUDE.md が自然に載る

方法2 — フェーズごとにセッションを分ける（最も確実）

  • 要件フェーズは docs/spec/ を cwd にして起動 → その CLAUDE.md がトップ＝毎ターン確実に載る
  • 実装コンテキストと混ざらない（実装時に要件の細部が載らない・逆も同様）

方法3 — 「やり方」は Skill / カスタムコマンドへ

  • ユーザーストーリーの型・受入基準・DB設計チェックリスト・ヒアリング観点などの方法論は実装の毎ターンには不要
  • requirements スキル（自動起動）か、.claude/commands/requirements.md（/requirements、$ARGUMENTS で対象機能を渡す）でフェーズの頭に注入

要件フェーズ用 CLAUDE.md の例（docs/spec/CLAUDE.md）

# 要件定義フェーズ（このフォルダ作業時のみ）

## 進め方
- この段階では実装・コード生成をしない（設計合意まで）
- 成果物は docs/spec/<チケットID>/requirements.md に書く

## 要件書フォーマット
- 背景 / ユーザーストーリー / 受入基準 / 非機能要件 / スコープ外

## 詰める観点（ヒアリング）
- 対象ユーザーと利用文脈 / 既存機能との整合 / 例外系・エラー時の挙動

## DB設計の規約
- テーブル名は複数形スネークケース / 論理削除は deleted_at
- created_at・updated_at を必ず持つ / 金額は整数（最小単位）で保持
効かせどころ：「この段階では実装しない」をレッドラインで明記しておくと、要件を詰めている途中でエージェントが勝手にコードを書き始める事故を防げる。ルート CLAUDE.md は薄く保つほど、フェーズ別ファイルの1行が効く。

5

ループの上限・停止条件

自走させる時は「いつ止まるか」を必ず先に決める。止まらない設計は事故のもと。

停止条件を書く（検証可能な形で）

  • /goal：検証可能な完了条件が真になるまで継続。毎ターン別の小モデルが達成を判定（作った本人に採点させない）。例：「test/auth が全 green かつ lint 0」
  • /loop：一定間隔で再実行（cron 的）。無限に回さないよう回数・時刻の枠を決める
  • Agent SDK で組む場合：maxTurns 等でターン上限を設定し、暴走を物理的に止める

並列時の事故防止

  • git worktree / --worktree で各エージェントを別チェックアウトに隔離（ファイル衝突の回避）
  • 並列数の上限は「自分がレビューしきれる数」に合わせる（レビュー帯域がボトルネック）
鉄則：停止条件・最大ターン・最大時間・並列数の4つは、走らせる前に必ず決める。

トークン / コストの上限はどこで設定するか

「使い方」ではなく「契約形態」で上限のかけ方が変わる。自分がどれで使っているかをまず確認する。

使い方上限のかけ方 / 確認場所
サブスク
(Pro / Max)		 ローリング5時間セッション上限＋週次上限で自動的に頭打ち（Pro＜Max5x＜Max20x）。使用中は /status・請求は /cost で確認。上限に達すると待機。
API 従量
(Console)		 Console でティア別の月間 spend cap。その範囲内で自分の spend limitを下げて設定可（ティア上限は超えられない）。使用量アラートも設定可。
コンテキスト窓 全モデル 200K トークン（一部 Enterprise は 500K）。長い作業は圧縮・分割前提で設計。
実行あたり ループ / SDK 側で最大ターン・最大時間を設定（→ 5章）。サブエージェントは各自が課金対象なので、使いどころを絞る。

コストを膨らませない実務ルール

  • 走らせる前に予算（時間・回数）を宣言してから始める
  • 大きなツール出力はファイルへ退避（context を汚さない＝無駄トークン削減）
  • 不要な MCP / ツールを外す（説明文が毎回課金対象の入力になる）
  • サブエージェントは「第二の意見が費用に見合う所」だけに投入

最初の30分でやること

  • ① CLAUDE.md を作り、コマンド・レッドライン・Done条件を10〜20行で書く（→ 1章）
  • ② .claude/settings.json に PostToolUse（lint / 型）と PreToolUse（危険コマンド block）を1つずつ（→ 2章）
  • ③ 使っている契約のコスト上限（サブスクの /cost or Console の spend limit）を一度見ておく（→ 6章）
  • ④ 毎回説明している手順が1つあれば .claude/skills/ に SKILL.md で切り出す（→ 3章）
  • ⑤ 要件定義など別フェーズがあれば、docs/spec/ に専用 CLAUDE.md を置き、場所で分ける（→ 4章）
  • ⑥ 自走は /goal に検証可能な停止条件を書いてから、小さなタスクで試す（→ 5章）
  • ⑦ 失敗が出たら、その場で CLAUDE.md か hook に1行足す（ラチェット）
